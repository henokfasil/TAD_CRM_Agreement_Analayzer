# MASTER CODEX PROMPT

## Build a Semi-Autonomous AI System for Mapping, Coding, Reviewing and Monitoring International Critical Raw Materials Agreements

You are acting as a senior AI systems architect, legal-NLP engineer, data engineer, full-stack Python developer, research software engineer and applied econometrician.

Your task is to design and implement a production-quality, semi-autonomous, human-in-the-loop research platform for analysing international agreements concerning critical raw materials, hereafter “CRM agreements.”

This is not a generic document chatbot.

This is a structured, auditable, reproducible policy-research system that must:

1. discover new CRM agreements from official sources;
2. ingest uploaded or downloaded documents;
3. extract and preserve text with page-level provenance;
4. classify candidate documents;
5. determine whether they meet project inclusion criteria;
6. segment agreements into legally and analytically meaningful provisions;
7. classify provisions according to a detailed CRM agreement codebook;
8. generate document-level and provision-level indicators;
9. provide evidence for every AI-generated code;
10. calculate confidence and uncertainty;
11. run independent verification and contradiction checks;
12. route ambiguous cases to human reviewers;
13. record reviewer decisions and adjudication;
14. maintain version history and an audit trail;
15. generate tables, visualisations and research-ready datasets;
16. allow comparison across agreements, countries, years and agreement types;
17. support semantic question answering with citations;
18. monitor implementation documents and follow-up actions;
19. export validated data for statistical and econometric analysis;
20. remain modular enough to extend to other international agreements later.

The working name of the platform is:

# CRM Agreement Intelligence System

Use this name throughout the project unless configuration overrides it.

---

# 1. CORE RESEARCH PRINCIPLE

The system must be AI-assisted, not AI-authoritative.

The AI may:

* discover;
* extract;
* summarise;
* classify;
* propose;
* compare;
* flag;
* score;
* generate draft interpretations.

The AI must not silently make final legal judgments.

Final validated decisions must be distinguishable from:

* raw model outputs;
* model confidence;
* rule-based checks;
* reviewer decisions;
* adjudicated decisions.

The system must make this distinction explicit in both the database and the user interface.

Never present AI-generated legal coding as verified fact unless a human reviewer has approved it or the configured validation policy allows auto-acceptance for low-risk variables.

# 7. CODEBOOK IMPLEMENTATION

Convert the paper’s coding framework into YAML.

Each variable must use a schema similar to:

```yaml
code: inv_spec_def
family: investment
label: Extensively defined CRM-related investment provision
description: >
  Indicates whether CRM-related investment is defined in substantial,
  operational or instrument-specific detail.
data_type: binary
allowed_values: [0, 1]
risk_level: medium
mandatory_human_review: false

dependencies:
  requires:
    inv_gen_ref: 1
    inv_crm: 1
    inv_basic_def: 1

positive_criteria:
  - names a financing mechanism
  - names an institution
  - identifies a specific programme
  - specifies a quantified amount
  - identifies concrete project-selection or implementation procedures

negative_criteria:
  - general promotion of investment
  - aspirational reference without mechanism
  - generic reference to private-sector participation
  - no CRM linkage

positive_examples:
  - text: "Participants intend to mobilise grants, guarantees, loans, or equity."
    explanation: "Specific financing instruments are listed."

negative_examples:
  - text: "The parties will encourage investment in critical minerals."
    explanation: "General intent without operational detail."

evidence_requirements:
  quotation_required: true
  location_required: true
  minimum_evidence_strength: 0.75
```

# 14. AGENT 5: LEGAL DESIGN AGENT

Analyse:

* legally binding versus non-binding;
* intention to be legally bound;
* mandatory versus best-efforts language;
* entry-into-force clauses;
* ratification;
* amendment;
* termination;
* dispute settlement;
* compliance;
* enforcement;
* private-state mechanisms;
* state-state mechanisms;
* consultation procedures;
* sequencing.

## Critical instruction

Do not infer bindingness solely from “shall,” “must,” or the document title.

Consider:

* explicit disclaimer clauses;
* governing law;
* ratification;
* entry into force;

# 16. NEGATIVE CODING PROTOCOL

A value of zero must not mean “the model did not notice anything.”

For negative codes, record:

* provisions searched;
* semantic queries used;
* keyword queries used;
* whether whole-document review occurred;
* confidence in absence;
* possible ambiguity.

Example:

```json
{
  "variable_code": "com_spec_fund",
  "proposed_value": 0,
  "confidence": 0.78,
  "negative_search_scope": {
    "whole_document_search": true,
    "keywords": ["USD", "EUR", "million", "billion", "grant", "loan", "equity"],
    "semantic_queries": [
      "quantified financial commitment",
      "specific funding amount for critical minerals"
    ],
    "candidate_passages_reviewed": 12
  },
  "reasoning_summary": "Investment is discussed, but no quantified financial commitment was identified."
}
```

Low-confidence negative findings must be reviewed.

# 42. FINAL BEHAVIOURAL INSTRUCTION

Be rigorous.

This is a policy-research system whose results may be used in institutional analysis. Prefer transparency over automation, explicit uncertainty over false certainty, and reproducibility over impressive-looking but unverifiable outputs.

Build the platform so that every important analytical claim can be traced back through:

```text
final validated code
→ human or adjudication decision
→ verifier output
→ initial AI proposal
→ cited provision
→ page-level source text
→ immutable original document
```

That traceability chain is the central design requirement of the entire system.
